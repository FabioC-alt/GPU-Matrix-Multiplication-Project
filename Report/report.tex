\begin{abstract}

Matrix multiplication ($C = A \times B$) serves as a fundamental building block in modern scientific computing, deep learning, and computer vision. 
Achieving peak theoretical performance on modern Graphics Processing Units (GPUs) requires overcoming severe memory bandwidth and instruction-level bottlenecks. 
This report details the implementation, optimization, and evaluation of various parallel matrix multiplication.
The first tests were executed are naive CPU implementations of Matrix Multiplication algorithms, while the last is the implementation of a CUDA specific multipication via the CuPy lib.
All the code is in Python. LLMs were used to support code and the report writing.




\end{abstract}

\section{Introduction}
\label{sec:intro}

Matrix multiplication is a foundational operation in modern computing, serving as the computational backbone for diverse applications ranging from climate modeling and scientific simulations to the training and inference of Large Language Models (LLMs).
Among these, the General Matrix Multiply (GEMM) operation, which computes the product of two dense matrices, is especially critical. Because of its high computational intensity, GEMM is the primary performance bottleneck in the linear algebra routines that underpin modern machine learning workloads.

To illustrate, the multiplication of two $2 \times 2$ matrices is defined as:
\[
\begin{bmatrix}
a & b \\
c & d
\end{bmatrix}
\begin{bmatrix}
x & y \\
z & w
\end{bmatrix}
=
\begin{bmatrix}
ax + bz & ay + bw \\
cx + dz & cy + dw
\end{bmatrix}
\]

While conceptually straightforward, a naive implementation of this algorithm scales with a time complexity of $\mathcal{O}(N^3)$ for $N \times N$ square matrices. This cubic complexity makes hardware-level optimization imperative for large-scale applications, particularly when attempting to fully saturate the massive parallel processing capabilities of modern Graphics Processing Units (GPUs). 

In this report, we investigate and compare several implementations of matrix multiplication. We begin with the naive CPU baseline and progressively introduce advanced optimization techniques, including shared memory tiling and memory-access optimizations such as matrix transposition.
\section{Implementation \& Optimization Strategies}
\label{sec:implementation}
This section forms the core technical contribution of your report. Describe each implementation variant step-by-step.

Given the interpreted nature of Python, we leverage the \texttt{Numba} library to JIT-compile our matrix multiplication functions for CPU execution.

To JIT-compile a function, we use the \texttt{@jit} decorator from Numba. This allows us to write Python code that is compiled to machine code at runtime, significantly improving performance.
\subsection{Baseline CPU Implementation}
The most straightforward approach to matrix multiplication is the naive algorithm, which utilizes three nested loops to compute the dot product of rows from matrix $A$ and columns from matrix $B$. While conceptually simple, this baseline implementation is highly inefficient on modern CPU architectures. Because matrix $B$ is accessed column-by-column, the memory access pattern violates spatial locality, resulting in frequent cache misses that severely bottleneck performance.
\begin{lstlisting}[language=Python, caption={Baseline Naive CPU Matrix Multiplication in Python}, label={lst:cpu_baseline}]
import numpy as np

@jit(nopython=True)
def naive_approach(A, B, C):
    for i in range(N):
        for j in range(N):
            for k in range(N):
                C[i][j] += A[i][k] * B[k][j]


\end{lstlisting}

\subsection{Exploiting the Locality principle}
Using the locality principle, we can improve cache performance by accessing data in a more cache-friendly manner. 
CPUs pre-load the data into cache line by line, in the naive implementation the A matrix is accessed row-wise, while the B matrix is accessed column-wise, which can lead to cache misses.
It is then possible to invert the order of the loops to access the B matrix row-wise, which is more cache-friendly and can lead to better performance.

\begin{lstlisting}[language=Python, caption={Baseline Naive CPU Matrix Multiplication with Locality Principle in Python}, label={lst:cpu_locality}]
@jit(nopython=True)
def exploiting_locality(A, B, C):
    for i in range(N):
        for k in range(N):
            for j in range(N):
                C[i][j] += A[i][k] * B[k][j]
\end{lstlisting}

\subsection{Transposed Approach}
In the transposed approach, we first transpose the second matrix B before performing the multiplication.
This allows us to access both matrices in a row-wise manner, which is more cache-friendly and can lead to better performance.
\begin{lstlisting}[language=Python, caption={Baseline Naive CPU Matrix Multiplication with Locality Principle in Python}, label={lst:cpu_locality_1}]
@jit(nopython=True)
def transposed_approach(A, B, C):
        for i in range(N):
            for j in range(N):
                for k in range(N):
                    C[i][j] += A[i][k] * B[j][k]

## In the main code
B_t = np.transpose(B).copy()
transposed_approach(A, B_t, C)

\end{lstlisting}

\subsection{Tiled Matrix Multiplication (Shared Memory)}
In Tiled Matrix Multiplication, we divide the matrices into smaller sub-matrices (tiles) that fit into the shared memory of the GPU.
This approach reduces the number of global memory accesses, which are slower than shared memory accesses, and allows for better utilization of the GPU's parallel processing capabilities.
\begin{lstlisting}[language=Python, basicstyle=\ttfamily\scriptsize, caption={Tiled Matrix Multiplication}, label={lst:tiling}]
@jit(nopython=True)
def tiling_approach(A, B, C, block_size):
    for ih in range(0, N, block_size):
        for kh in range(0, N, block_size):
            for jh in range(0, N, block_size):
                # min in necessary in case the matrix is smaller than the sum of the 
                # the block and the already reached position
                for i in range(ih, min(ih + block_size, N)):
                               for k in range(kh, min(kh + block_size, N)):
                                              for j in range(jh, min(jh + block_size, N)):
                                                             C[i][j] += A[i][k] * B[k][j]
\end{lstlisting}

\subsection{GPU Optimization with CuPy}
Specialized GPU libraries like CuPy can be used to perform matrix multiplication on the GPU with minimal code changes.
\begin{lstlisting}[language=Python, basicstyle=\ttfamily\scriptsize, caption={CuPy Implementation}, label={lst:cupy}]
import cupy as cp  

A = np.random.rand(N, N).astype(np.float32)
B = np.random.rand(N, N).astype(np.float32)
C = np.zeros((N, N), dtype=np.float32)

C = cp.dot(A, B)
cp.cuda.Stream.null.synchronize()

\end{lstlisting}

To visualize these performance trends, \textit{Figure 1} and \textit{Figure 2} illustrate the execution times for all tested approaches under JIT and non-JIT compilation, respectively. Ultimately, the tiling approach with a block size of 64 stood out as the most effective configuration. By optimizing cache locality, this specific setup maximized CPU efficiency to the point of rivaling the dedicated GPU CuPy implementation, making a block size of 64 the optimal choice for these experiments.
